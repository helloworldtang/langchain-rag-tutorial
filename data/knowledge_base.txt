# LangChain RAG 实战指南

## 什么是 RAG（检索增强生成）

RAG 是 Retrieval-Augmented Generation 的缩写，中文叫"检索增强生成"。它是一种把"外部知识检索"和"大语言模型生成"结合起来的架构：先从一个知识库里检索出与用户问题最相关的文档片段，再把这些片段连同问题一起喂给大模型，让模型基于检索到的内容生成答案。

RAG 的标准工作流程是：用户提问 → 检索相关文档 → 把文档拼进提示词 → 大模型生成答案。本项目的实现流程为：加载文档 → 文本分割 → 向量嵌入与索引 → 相似度检索 → 拼接提示词 → LLM 生成答案。

### RAG 解决了 LLM 的三大痛点

1. **知识截止问题**：大模型的知识有训练截止日期，无法回答最新发生的事情。RAG 通过检索实时文档，可以让模型回答最新的问题。
2. **私有数据缺失**：大模型不知道你的个人笔记、企业内部文档。RAG 把这些私有数据接入模型，实现定制化问答。
3. **幻觉问题**：大模型可能会一本正经地编造不存在的答案。RAG 让模型基于真实检索到的文档生成，并能给出引用来源，减少幻觉、答案可追溯。

## LangChain 是什么

LangChain 是一个用于构建大语言模型（LLM）应用的开源框架，核心理念是"通过可组合性（composability）构建 LLM 应用"。它把 LLM 应用拆成一组可拼装的标准组件，开发者像搭积木一样组装出文档问答、聊天机器人、智能代理等应用。本项目使用 LangChain 构建一个完整的 RAG 问答系统。

LangChain 采用分包架构：`langchain-core` 提供核心抽象（Document、Runnable 等基础接口），`langchain` 提供链与组合语法（LCEL），`langchain-community` 汇集社区集成，各类第三方集成则拆成独立的轻量包——例如本项目用到的 `langchain-ollama`（Ollama 的 LLM 与 Embedding 集成）和 `langchain-text-splitters`（文本分割）。这种设计让你只安装真正用到的部分，依赖更轻量。

## LangChain 的核心组件

### 1. Document（文档对象）

Document 是 LangChain 中表示一段文本的标准对象，包含两个字段：`page_content`（文本内容）和 `metadata`（元数据字典，如来源、页码等）。所有文档加载、分割、检索的流转都以 Document 为基本单位。

### 2. Text Splitters（文本分割器）

大模型的上下文窗口有限，整篇长文档无法直接用于检索。Text Splitters 把长文档切分成一个个较小的语义块（chunk）。本项目使用 `RecursiveCharacterTextSplitter`：它按一组分隔符列表（段落 `\n\n` → 换行 `\n` → 中文句末标点 `。！？；` → 逗号 `，` → 空格 → 单字）逐级递归切分，尽量在自然语义边界断开；用 `chunk_size` 控制每块最大长度、`chunk_overlap` 控制相邻块之间的重叠（避免把完整句子从中间切断）。相比只在单一分隔符（如 `\n\n`）处切的 `CharacterTextSplitter`，递归切分对没有空格分隔的中文更友好、块边界更自然，是 LangChain 官方推荐的通用切分器。

### 3. Embeddings（嵌入模型）

Embeddings 负责把文本转换成高维向量（vector），让语义相近的文本在向量空间中距离也接近。LangChain 定义了统一的 `Embeddings` 接口，包含两个方法：`embed_query(text)` 把查询转成向量，`embed_documents(texts)` 把一批文档转成向量。本项目用 Ollama 提供的 `nomic-embed-text` 模型生成嵌入。

### 4. VectorStore（向量数据库）

VectorStore 用来存储文档向量并提供相似度检索能力。LangChain 支持多种向量库，如 FAISS、Chroma、Milvus、Pinecone 等。本项目使用 FAISS（faiss-cpu），它是一个高效的稠密向量检索库，支持本地持久化：把索引保存到磁盘后，下次启动直接加载，避免重复计算嵌入。

### 5. LLM（大语言模型）

LangChain 用 `ChatModel` 接口统一封装各类大模型，调用 `.invoke()` 方法即可得到回复。本项目通过 `langchain-ollama` 集成包使用 `ChatOllama`，在本地通过 Ollama 运行 `deepseek-r1:1.5b` 模型，无需联网、无需 API Key。

### 6. Retriever（检索器）

Retriever 是 LangChain 中"检索"的抽象接口，输入查询、返回相关 Document。VectorStore 可以通过 `.as_retriever()` 转成 Retriever，再与 LLM 组合成链。本项目为了演示混合检索细节，直接调用 VectorStore 的 `similarity_search` 方法，而非走 Retriever 抽象。

## 相似度检索：稠密检索（FAISS）

稠密检索（dense retrieval）基于向量相似度工作：把查询和文档都转成稠密向量，然后计算向量距离（如余弦相似度、欧氏距离），距离越近代表语义越相关。本项目用 FAISS 做稠密检索，调用 `similarity_search_with_score` 同时拿到文档和相似度分数。

稠密检索的优势是"懂语义"：查询里写"汽车"，它能检索出写"轿车""小轿车"的文档，即使字面完全不同。它擅长理解同义词、近义表达和整体语义关联。

## 关键词检索：稀疏检索（BM25）

稀疏检索（sparse retrieval）基于词项匹配工作，代表算法是 BM25。BM25 本质是一个改进的 TF-IDF：它对文档建立倒排索引，根据查询词在文档中出现的频率（TF）、词的稀有程度（IDF）以及文档长度归一化，给每个文档打分。本项目用 `rank-bm25` 库的 `BM25Okapi` 实现稀疏检索。

稀疏检索的优势是"精确匹配"：当查询包含专有名词、人名、产品型号、代码标识符等精确关键词时，BM25 往往比向量检索更准。需要注意的是，BM25 依赖分词，中文需要先用 jieba 等工具分词（英文按空格切分即可），否则检索效果会明显下降。

## 混合检索与 RRF 融合（本项目特色）

稠密检索和稀疏检索各有长短：FAISS 懂语义但可能漏掉精确关键词，BM25 精确匹配但不懂同义词。混合检索（hybrid search）把两者结果合并，取长补短，覆盖更全面的检索场景。

多路检索的结果如何合并？本项目使用 RRF（Reciprocal Rank Fusion，倒数排名融合）算法。RRF 不关心各检索器给出的原始分数（不同检索器分数量纲不同，不可直接相加），只看每条文档在各检索器中的排名：对排名为 r 的文档，贡献分数 `1 / (k + r)`（k 通常取 60），然后把它在所有检索器中的分数累加，最后按总分重新排序。排名越靠前、被越多检索器同时命中的文档，最终分数越高。

RRF 的优点是简单、鲁棒、无需调参归一化，是工业界混合检索最常用的融合方法之一。

## LangChain 表达式语言（LCEL）

LangChain 推荐用 LCEL（LangChain Expression Language）把组件拼装成链，用管道符 `|` 连接，例如 `retriever | prompt | llm | output_parser`。LCEL 链天然支持流式输出、异步、批处理和可观测性。本项目为了让混合检索和 RRF 融合逻辑清晰可见，手动实现了检索与拼接，没有使用完整的 LCEL 链；在生产实践中，可以用 LCEL 把官方 Retriever、PromptTemplate、ChatModel 和 StrOutputParser 组合成标准的 RAG 链。

## 应用场景

1. **个人知识库问答**：把笔记、文章、文档变成可对话的知识库。
2. **企业内部搜索**：基于私有文档构建企业问答系统，员工自助查询。
3. **客服机器人**：基于产品手册、FAQ 生成准确回复，减少幻觉。
4. **代码库助手**：基于代码和文档回答技术问题。
5. **研究文献助手**：帮研究者快速从大量论文中检索并总结相关内容。

## 扩展方向

1. **更多数据源**：接入 PDF、Word、网页、数据库等。
2. **多文档管理**：支持多个知识库的检索与路由。
3. **多轮对话**：引入聊天历史（memory），支持上下文连贯的问答。
4. **重排序（Rerank）**：在检索后用交叉编码器对结果精排，提升 Top-K 质量。
5. **Web UI 与 API**：用 Gradio/Streamlit 做界面，或用 FastAPI 部署成 REST 服务。
